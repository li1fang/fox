# 训练运行界面设计

## 定位

训练运行界面是 fox 健身模块的第一执行界面。

它采用本地 Web Runtime UI，而不是传统终端 TUI。

目标不是做完整 iPad App，而是在浏览器里验证真正的训练体验：

- 看得清。
- 点得快。
- 动画能带节奏。
- 倒计时可靠。
- 反馈流程顺。
- 状态机边界清楚。

## 运行方式

第一版运行方式：

```text
local browser UI <-> local backend/state machine
```

UI 职责：

- 展示当前训练状态。
- 播放节奏动画。
- 展示倒计时。
- 接收用户点击。
- 展示 AI 生成的选项和建议。

核心程序职责：

- 保存 WorkoutSession。
- 推进状态机。
- 计算倒计时状态。
- 记录 FeedbackEvent。
- 应用 Adjustment。
- 调用 AI 生成计划、选项、建议和总结。

UI 不直接修改训练事实。

## 主屏布局

训练中主屏分为四个区域：

```text
┌─────────────────────────────────────────┐
│ 今日主训练进度                           │
│ 当前动作 / 当前组 / 剩余主训练组数        │
├─────────────────────────────────────────┤
│                                         │
│        节奏演示区                        │
│        方块放大 / 缩小 / 顶峰震动         │
│                                         │
├─────────────────────────────────────────┤
│ 倒计时圆环或进度条                       │
│ 剩余秒数 / 当前阶段                      │
├─────────────────────────────────────────┤
│ 完成本组                                │
│ 紧急停止                                │
└─────────────────────────────────────────┘
```

必须常驻显示：

- 当前动作名。
- 当前组，例如 `第 2 / 3 组`。
- 当前目标，例如 `7.5kg x 10` 或 `60 秒`。
- 今日主训练进度，不包括拉伸。
- 可视化倒计时。

## 节奏演示

节奏演示是训练执行界面的核心。

第一版使用一个几何方块：

- 放大：发力或收缩阶段。
- 缩小：还原或离心阶段。
- 短震动：顶峰收缩。
- 静止：保持。

节奏由 tempo script 驱动。

示例：

```json
{
  "tempo": [
    { "phase": "lift", "seconds": 2, "visual": "expand" },
    { "phase": "peak", "seconds": 1, "visual": "vibrate" },
    { "phase": "lower", "seconds": 4, "visual": "shrink" }
  ]
}
```

如果动作没有明确 tempo，则使用默认呼吸节奏：

- 2 秒发力。
- 1 秒顶峰。
- 3 秒还原。

## 倒计时

倒计时分为两类：

- work timer：计时型动作，例如平板支撑。
- rest timer：组间休息。

重复次数型动作可以没有 work timer，但仍显示节奏演示。

倒计时显示要求：

- 圆环优先。
- 窄屏或实现简化时可用进度条。
- 同时显示剩余秒数。
- 使用真实时间戳计算剩余时间。
- 页面恢复可见时从后台重新同步。

## 倒计时打断规则

常规倒计时不能被打断。

用户不能在一组中途频繁评价，以免训练流程碎裂。

唯一例外是紧急停止：

- 疼痛。
- 眩晕。
- 抽筋。
- 器材危险。
- 其他明显不适。

紧急停止必须常驻可见，但视觉上弱于“完成本组”，避免误触。

## 完成本组流程

用户点击 `完成本组` 后进入反馈屏。

第一步：记录实际完成。

根据动作类型显示：

- 次数输入：实际完成几次。
- 时长输入：实际坚持几秒。
- 重量输入：是否使用计划重量。

第二步：选择反馈。

固定选项：

- 正常完成。
- 没跟上。
- 太重了。
- 太轻了。
- 速度不稳定。
- 不确定动作是否正确。
- 感觉疼痛或不舒服。
- 跳过或提前停止。

第三步：AI 生成动作相关常见问题。

AI 可根据当前动作和历史记录补充 2-4 个选项，例如：

- 肩推：肩膀顶端不舒服。
- 侧平举：借力太多。
- 单杠：第二次后拉不动。
- 上斜推举：下降阶段失控。

第四步：自由备注。

用户可以不填。

## “没跟上”处理

没跟上不是固定降级规则能完全解决的问题。

系统必须把足够上下文交给 AI 和规则层：

- 当前动作。
- 当前组序号。
- 这是第一组、中间组还是最后一组。
- 当前训练已完成比例。
- 最近同动作表现。
- 本次实际完成次数。
- 原计划目标。
- 用户备注。
- 是否有疼痛。
- 今日 check-in 状态。

处理原则：

- 第一组没跟上：倾向整体降低该动作后续强度。
- 中间组没跟上：倾向降低后续组或延长休息。
- 最后一组没跟上：可能只记录疲劳，不必改计划。
- 举不动：降低重量、减少次数或结束该动作。
- 疼痛：停止该动作，不做同部位高风险替代。

AI 输出 adjustment suggestion，核心程序校验后应用。

## 休息屏

休息屏显示：

- 休息倒计时。
- 下一组预告。
- 上一组记录。
- 简短提示或鼓励。

允许操作：

- 延长休息 30 秒。
- 准备好了，进入下一组。
- 紧急停止。

休息可以被用户提前结束，因为它不属于正在发力的一组。

## 今日主训练进度

今日主训练进度不包括拉伸。

默认包含：

- 主要力量动作。
- 核心训练。
- 有氧主训练。

默认不包含：

- 热身。
- 冷却。
- 拉伸。

进度计算：

```text
已完成主训练组数 / 计划主训练组数
```

计时型动作按一组计算。

如果动作被跳过：

- 计入已处理。
- 不计为完成。
- 在总结中单独显示。

## 调整建议屏

当系统决定调整计划时显示。

内容：

- 触发原因。
- 调整前。
- 调整后。
- 简短解释。

示例：

```text
你刚才第 1 组只完成 6 / 10 次。
下一组从 10 次降到 8 次，休息延长到 120 秒。
目标是保持动作稳定，不硬顶。
```

用户可以：

- 接受调整。
- 手动修改。
- 跳过当前动作。

## 总结确认屏

训练结束后进入总结确认。

总结包含：

- 今日训练重点。
- 完成动作和组数。
- 未完成或跳过内容。
- 调整原因。
- 疼痛或不适记录。
- 下次建议。

用户可以：

- 确认总结。
- 编辑备注。
- 标记总结不准确。

只有确认后，WorkoutSession 才写入 confirmed Entry。

## 今日状态 check-in

开始训练前先做轻量 check-in。

字段：

- 睡眠：好 / 一般 / 差。
- 疲劳：低 / 中 / 高。
- 饥饿：不饿 / 有点饿 / 很饿。
- 压力：低 / 中 / 高。
- 疼痛：无 / 肩 / 腰 / 膝 / 其他。
- 可训练时长。

这些数据未来可以被睡眠、饮食、财务模块自动补充。

第一版手动输入即可。

## 视觉原则

- 深色或中性背景。
- 当前动作和倒计时最大。
- 按钮足够大，适合 iPad 触控。
- 每屏只突出一个主操作。
- 节奏动画要稳定、克制、可关闭。
- 不使用复杂装饰。
- 不做 3D 形象。

## 最小页面

第一版需要 6 个页面状态：

1. 今日状态 check-in
2. 计划确认
3. 主训练执行
4. 组后反馈
5. 休息倒计时
6. 总结确认

## 最小事件

UI 只发送事件：

- `check_in_submitted`
- `plan_accepted`
- `plan_rejected`
- `set_finished`
- `feedback_submitted`
- `rest_extended`
- `rest_skipped`
- `emergency_stop`
- `summary_confirmed`
- `summary_corrected`

## 数据落库原则

必须落库：

- 用户 check-in。
- 计划。
- 每组实际完成。
- 用户反馈。
- 计时开始和结束。
- 调整建议和最终调整。
- 训练总结。
- 用户确认状态。

只做临时 UI 状态：

- 当前动画帧。
- 当前按钮 hover/pressed 状态。
- 未提交的反馈表单。
- 局部视觉倒计时插值。

## 第一版验收

一次训练闭环通过即为合格：

1. 用户完成 check-in。
2. 系统展示计划。
3. 用户接受计划。
4. UI 展示节奏方块和倒计时。
5. 用户完成一组并提交反馈。
6. 系统根据没跟上或太重调整后续计划。
7. 用户完成或提前结束训练。
8. 系统生成总结。
9. 用户确认总结。
10. 训练记录进入 confirmed 状态。
